## [발표자 A — 약 5분]

### [오프닝]

안녕하세요. 이번 발표에서는 김준서, 임정현 두명이 개발한 **무강대학교 AI 학사행정 서비스**를 소개하겠습니다.

이 프로젝트는 FastAPI와 Vanilla JavaScript를 기반으로 한 웹 애플리케이션으로,
학생과 교직원이 실제 학사 업무를 처리할 수 있도록 설계했습니다.

저는 서비스 기능과 AI·데이터베이스 레이어를 담당했고,
발표자 B가 AWS 인프라와 배포 파이프라인을 담당했습니다.

---

### [1. 주요 기능 소개]

이 서비스의 핵심 기능은 크게 네 가지입니다.

첫째, **학번/사번 기반 로그인**입니다. 최초 로그인 시 비밀번호를 변경하는 First Setup 절차가 포함되어 있습니다. 인증은 JWT와 bcrypt로 처리합니다.

둘째, **수강신청 시스템**입니다. 강의 조회, 장바구니 담기, 수강신청 및 취소가 가능하고, 정원 초과 시 대기열 처리도 구현했습니다.

셋째, **관리자 기능**입니다. 과목 생성·삭제, PDF 강의계획서 업로드, 수강신청 일정 관리를 관리자 페이지에서 처리할 수 있으며, 학생 통계와 관리자 통계 API도 제공합니다.

넷째, **RAG 기반 AI 챗봇**입니다. 학사 규정 문서를 벡터 검색하여 학생 질문에 답변하고, AI 과목 추천 기능도 제공합니다.

---

### [2. 기술 스택]

백엔드는 **Python 3.11 + FastAPI**를 사용했고, SQLAlchemy ORM으로 데이터베이스를 관리합니다.

프론트엔드는 Vanilla HTML/CSS/JavaScript로 구성했으며, Nginx가 정적 파일을 서빙합니다. 별도 프레임워크를 사용하지 않아 의존성을 최소화했습니다.

데이터베이스는 **PostgreSQL**을 메인으로 사용합니다. **pgvector 확장**을 붙여 RAG 벡터 검색까지 PostgreSQL 안에서 처리했습니다. 로컬 개발 시에는 SQLite로 fallback도 가능합니다.

AI 기능은 **AWS Bedrock**으로 처리합니다.
- 임베딩 생성에는 `amazon.titan-embed-text-v1`(1536차원)을 사용하고,
- 답변 생성에는 `anthropic.claude-v2`를 사용합니다.

챗봇 질의 흐름은 사용자 질문이 들어오면, FastAPI가 PostgreSQL에서 관련 문서를 벡터 검색하고, 그 문맥을 Bedrock에 전달해 답변을 생성한 뒤 반환하는 구조입니다.

인프라는 Docker와 Terraform으로 구성했으며, GitHub Actions로 CI/CD를 자동화했습니다. 자세한 내용은 발표자 B가 이어서 설명하겠습니다.

## [발표자 B — 약 5분]

### [3. AWS 인프라 아키텍처]

AWS 인프라를 설명하겠습니다.

서울 리전(`ap-northeast-2`)에 VPC(`10.0.0.0/16`)를 구성했습니다.

**네트워크 구성**은 퍼블릭 서브넷과 프라이빗 서브넷으로 분리했습니다.
- 퍼블릭 서브넷(AZ-a)에는 Nginx 프록시 EC2(`t3.micro`)와 NAT Gateway를 배치했습니다. 프록시 EC2의 퍼블릭 IP가 `43.201.54.86`으로 외부 트래픽의 진입점입니다.
- 프라이빗 서브넷(AZ-a/c)에는 실제 앱 서버인 Blue/Green ASG EC2(`t3.medium`)를 배치했습니다. Private 서브넷의 EC2가 ECR·Bedrock과 통신할 때는 NAT Gateway의 EIP를 통해 아웃바운드합니다.

**Blue/Green 배포 구조**입니다.
현재는 Blue ASG가 활성 상태이고, Green ASG는 대기 중입니다. 배포 시 프록시 Nginx의 업스트림을 Blue에서 Green으로 전환하는 방식으로 무중단에 가깝게 운영합니다. 전환 명령은 SSM RunCommand로 프록시 EC2에 원격 실행합니다.

**데이터 레이어**입니다.
RDS PostgreSQL은 프라이빗 구조라 외부 직접 접근이 불가능하고, 운영 점검 시에는 SSM 포트포워딩으로 접속합니다.

컨테이너 이미지는 ECR의 `mugang-backend`, `mugang-frontend` 두 리포지토리로 관리하고, Terraform 상태는 S3 백엔드로 관리합니다.

---

### [4. CI/CD 파이프라인]

GitHub Actions를 통해 `main` 브랜치에 push가 발생하면 자동으로 배포가 진행됩니다.

흐름은 다음과 같습니다.
- Frontend/Backend Docker 이미지 빌드
- ECR에 이미지 푸시
- Terraform으로 Blue/Green 색상 전환
- SSM 명령으로 Proxy Nginx 업스트림 스위칭

전체 리소스를 정리할 때는 별도 `destroy.yml` 워크플로를 수동으로 트리거합니다.

---

### [5. DynamoDB에 대해]

아키텍처 다이어그램에 DynamoDB가 표시되어 있는데, 현재는 실제로 연결해 사용하고 있지 않습니다.

챗봇 대화 이력을 저장할 목적으로 Terraform에 `mugang-chat-history` 테이블(`hash_key=session_id`, `range_key=timestamp`)을 미리 정의해 두었습니다. 현재는 소규모 운영이라 PostgreSQL로 충분하고, 연결 시 추가 비용이 발생하므로 트래픽이 늘어나는 시점에 붙일 예정입니다. 현재 챗봇 이력은 PostgreSQL에 저장되며 기능적으로는 완전히 동작합니다.

---

### [6. 향후 개선 계획]

네 가지 개선 방향을 잡고 있습니다.

- **DynamoDB 연결**: 챗봇 이력 저장을 PostgreSQL에서 DynamoDB로 이전하여 비용 효율적인 NoSQL 구조로 전환
- **Alembic 마이그레이션**: 현재 `create_all` 방식에서 버전 관리 가능한 마이그레이션으로 전환
- **테스트 자동화**: pytest 기반 API 시나리오 테스트 구축
- **배포 표준화**: K8s 매니페스트와 Terraform 운영 모델 중 하나로 통일

이상으로 발표를 마칩니다. 감사합니다.
