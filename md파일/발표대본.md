## 시작 멘트

안녕하세요, 저희는 대학교 수강신청 플랫폼을 개발한 무강이팀입니다.
저희 무강이팀은 김준서, 임정현이 한팀으로 개발을 진행하였습니다. 
지금부터 README 번호 순서대로 발표를 시작하겠습니다.

---

## 1) 주요 기능

첫 번째는 로그인과 초기 비밀번호 변경입니다.  
학생은 학번, 교직원은 사번으로 로그인합니다.

두 번째는 수강신청 기능입니다.  
강의 조회, 장바구니, 신청, 취소, 정원 초과 시 대기열 처리까지 지원합니다.

세 번째는 관리자 기능입니다.  
과목 생성/삭제, PDF 업로드, 수강신청 일정 관리를 제공합니다.

네 번째는 AI 기능입니다.  
RAG 기반 질의응답과 과목 추천 API를 제공합니다.

---

## 2) 테스트 계정

점검 계정은 README 기준으로 다음과 같습니다.

학생 계정: `201811047 / 123123123` 학생 계정은 학번으로 이루어진 계정으로 
교직원 계정: `staff / 123123123` 교직원 계정은 부여된 아이디로 이루어진 계정으로 진행합니다.

환경에 따라 first-setup 요구 여부는 달라질 수 있습니다.

---

## 3) 기술 스택

백엔드는 Python 3.11, FastAPI, SQLAlchemy(에스-큐-앨 알케미), Uvicorn입니다.

프론트엔드는 Vanilla HTML/CSS/JS와 Nginx입니다.

인증은 JWT + bcrypt(비크립트)입니다.

DB는 PostgreSQL이 메인이고, 로컬 fallback으로 SQLite를 둡니다.

AI는 AWS Bedrock을 사용합니다.

인프라는 Docker, Terraform, AWS이고  
CI/CD는 GitHub Actions입니다.

---

## 4) 저장소 구성 요약

구조는 크게 네 영역입니다.

백엔드 API 계층이 인증, 수강신청, 관리자, AI 요청을 처리하고,  
데이터 모델 계층이 사용자와 강의, 수강 정보 구조를 관리합니다.

프론트엔드에서는  
학생 화면 로직과 챗봇 상호작용 로직이 분리되어 있고,  
인프라 영역에서 Terraform과 배포 자동화 파이프라인을 관리합니다.

---

## 5) 환경 구분

local은 개인 개발용입니다.  
aws-dev는 통합 검증용입니다.  
aws-prod-like는 운영 시뮬레이션용입니다.

핵심은 운영 배포 기준이 Terraform + Blue/Green이라는 점입니다.

---

## 6) 로컬 실행

로컬 시작 명령은 하나입니다.

`docker compose up -d --build`

확인 주소는 세 가지입니다.

- 먼저 프론트 화면이 정상적으로 열리는지 확인합니다.
- 다음으로 API 문서 페이지가 열리는지 확인합니다.
- 마지막으로 헬스체크 응답이 정상인지 확인합니다.

---

## 7) 백엔드 단독 실행

옵션으로 백엔드만 띄울 수도 있습니다.

의존성 설치 후 Uvicorn으로 8000 포트를 실행하면 되고,  
개발 중 API 동작만 빠르게 확인할 때 유용합니다.

---

## 8) AWS 인프라 실행

Terraform으로 init, plan, apply 순서로 올립니다.

그리고 output에서  
`proxy_public_ip`, `rds_endpoint`, `active_color`를 확인합니다.

지금 화면은 aws 스냅샷 및 리소스 정리 화면입니다.
---

## 9) CI/CD

main 브랜치 push 시 자동 배포됩니다.

흐름은,

1. 이미지 빌드  
2. ECR 푸시  
3. Terraform 기반 Blue/Green 전환  
4. Proxy Nginx 업스트림 스위칭

입니다.

---

## 10) 주요 API 엔드포인트

핵심 API는 다음 네 묶음입니다.

- Auth: 로그인, first-setup
- Lecture/Enrollment: 강의 조회, 수강신청/취소
- Student/Admin: 통계와 운영 기능
- AI: 챗봇 질의응답, 과목 추천

---

## 11) 아키텍처 다이어그램

요약하면,

사용자 요청은 Proxy EC2로 들어오고,  
Blue/Green 앱 서버로 라우팅됩니다.

앱은 RDS PostgreSQL을 사용하고,  
AI 질의는 Bedrock과 연동됩니다.

즉, 서비스 구조와 인프라 구조를 분리해 안정적으로 운영합니다.

---

## 12) 주요 경로

발표 중 코드 질문이 나오면,  
백엔드 API 처리부, 프론트 메인 로직, 인프라 정의, 배포 자동화 순서로 확인하면 됩니다.

즉, 기능 이슈는 API와 프론트에서 확인하고,  
운영 이슈는 인프라와 배포 파이프라인에서 확인하는 방식입니다.

---

## 13) 운영 체크 포인트

배포 후에는 세 가지를 우선 확인합니다.

- 서비스가 정상 응답하는지 헬스체크로 먼저 확인합니다.
- API 문서 화면이 열리는지 확인해서 엔드포인트 상태를 점검합니다.
- 현재 활성 배포 색상이 무엇인지 확인해서 Blue/Green 전환 상태를 검증합니다.

## 14) 향후 개선 후보

향후 계획은 세 가지입니다.
